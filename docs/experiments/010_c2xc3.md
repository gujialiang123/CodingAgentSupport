# Experiment 010 — C2 × C3 (helper tests × gates) 2×2

**Plan priority 4.** Isolate the individual and joint effects of C2 (helper
reproduction tests) and C3 (validation gates), now that P1 (container execution)
and P2 (container-validated, frozen T3/T4 helpers) make both supports real.

## 1. Design

2×2 factorial, one factor per axis:

|            | no Gates | Gates |
|------------|----------|-------|
| no Helper  | C0_minimal | C3_gates |
| Helper     | C2_tests   | C2_C3 |

- **Cohort:** `data/tasks/ablation_t34.jsonl` — the **7/12 tasks** whose C2 helper
  is class **T3/T4** (container-validated fail-before + pass-after-gold), frozen
  in `data/helpers/`. Using only T3/T4 tasks removes the confound of invalid
  helpers so C2's effect is measured where the helper actually reproduces the bug.
- **Seeds:** 0,1,2 (3 seeds).
- **Runs:** 7 tasks × 4 conditions × 3 seeds = **84 runs**.
- **Budget:** 25 turns, in-container (`--in-container`), qwen3.7-plus, max_tokens 4096.
- **Helpers:** loaded read-only from the frozen cache (`--helper-cache-dir
  data/helpers`); no per-run regeneration, so the exact validated T3/T4 helper is
  what the agent sees, and C2/C2_C3 are deterministic w.r.t. the helper.

The 7 T3/T4 tasks: psf/requests-1142 (T4), psf/requests-2931 (T3),
pytest-7571 (T4), pytest-10051 (T4), pylint-4604 (T3), flask-5014 (T4),
pydata/xarray-3677 (T4).

## 2. Questions this cleanly answers

1. Do **helper tests** (C2) raise resolution / F2P over C0?
2. Do **gates** (C3) raise resolution or mainly cut P2P regressions over C0?
3. Are helper and gates **complementary** (C2_C3 > max(C2,C3)) or redundant?
4. Does C2 raise F2P but induce **test overfitting** (helper passes, official fails)?

## 3. Test-generation feasibility (P2 result, reported, not dropped)

From freezing helpers on the full 12-task cohort (qwen3.7-plus, k=3, in-container):

| Class | n | meaning |
|-------|---|---------|
| T4_decoupled_valid | 5 | T3 + no leakage/overfit audit flag |
| T3_valid_reproduction | 2 | fails on base, passes on gold |
| T2_incompatible_oracle | 2 | fails on base AND on gold |
| T1_non_reproducing | 1 | passes on base (no repro) |
| T0_invalid | 2 | cannot collect/execute |

**7/12 (58%) usable (T3/T4).** This is itself a headline feasibility number:
a blind, gold-free generator produces a container-valid reproduction test for the
majority of issues, but ~42% are unusable — so C2 must be reported conditional on
helper validity, which is exactly what this cohort does.
